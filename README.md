# WarehouseInventory

Мы сделали систему управления складом с несколькими клиентами:

- `WarehouseInventory.Api` — основной backend на ASP.NET Core Web API
- `WarehouseInventory.Desktop` — desktop-клиент на WPF
- `WarehouseInventory.Web` — MVC web-приложение
- `WarehouseInventory.Frontend` — отдельный frontend на React + Vite
- `WarehouseInventory.Tests` — автотесты для backend-сервисов

Проект рассчитан на учет товаров на складе, авторизацию пользователей, работу с инвентаризацией, загрузку файлов к карточкам товаров и формирование простых отчетов.

## Общее описание

`WarehouseInventory` — проект по автоматизации складского учета.

Основная идея системы:

- хранить карточки товаров;
- выполнять авторизацию через JWT;
- получать и изменять остатки через API;
- прикреплять файлы к товарам;
- проводить инвентаризацию;
- получать сводный отчет по результатам проверки.

Основная бизнес-логика сосредоточена в `WarehouseInventory.Api`, а desktop-клиент выступает как полноценный интерфейс оператора/администратора.

## Архитектура

Система разделена на независимые модули:

1. `WarehouseInventory.Api`
   Основной backend и источник данных.

2. `WarehouseInventory.Desktop`
   WPF-клиент, который взаимодействует с API и реализует основные пользовательские сценарии по складу.

3. `WarehouseInventory.Web`
   Отдельный ASP.NET Core MVC проект с локальными формами.

   Отдельный frontend на React/Vite.

5. `WarehouseInventory.Tests`
   Набор unit/integration-style тестов для сервисов backend-части.

## Роли и сценарии работы

В desktop-клиенте предусмотрены пользовательские сценарии для двух типов доступа:

### Администратор

- просмотр товаров;
- добавление и редактирование товаров;
- удаление товаров;
- работа с API-сессией;
- доступ к вкладке остатков;
- доступ к отчетам.

### Оператор

- просмотр товаров;
- ручное добавление и импорт;
- доступ к вкладке остатков;
- проведение инвентаризации;
- доступ к отчетам.

### Сценарий инвентаризации

1. Пользователь открывает вкладку `Остатки`.
2. Сканирует товар по `SKU` или выбирает его из списка.
3. Вводит фактическое количество.
4. Приложение сравнивает факт с ожидаемым остатком.
5. Назначается статус:
   - `совпадает`
   - `недостача`
   - `избыток`
6. После проверки всех позиций пользователь нажимает `Сохранить`.
7. Сводка становится доступна во вкладке `Отчеты`.

## API

Базовый путь:

```text
/api
```

### Авторизация

- `POST /api/auth/register` — регистрация
- `POST /api/auth/login` — вход

После успешной авторизации сервер возвращает:

- `token`
- `expiresAtUtc`
- `userName`
- `email`

### Товары

- `GET /api/inventory-items` — список товаров
- `GET /api/inventory-items/{id}` — получить товар по `Id`
- `POST /api/inventory-items` — создать товар
- `PUT /api/inventory-items/{id}` — обновить товар
- `DELETE /api/inventory-items/{id}` — удалить товар

### Файлы товаров

- `POST /api/inventory-items/{id}/files` — загрузить файл
- `GET /api/inventory-items/{id}/files` — получить список файлов
- `GET /api/inventory-items/{id}/files/{fileId}` — скачать файл

### Защита API

Кроме `/api/auth/*`, методы требуют JWT-токен в заголовке:

```text
Authorization: Bearer <token>
```

## Desktop-клиент

Основные разделы рабочего окна:

- `Обзор`
- `Товары`
- `Остатки`
- `Отчеты`
- `Профиль`

### Что делает раздел `Товары`

- показывает список складских позиций;
- позволяет выбрать товар и отредактировать поля;
- позволяет вручную создать новую карточку;
- импортирует товары из `CSV`;
- может отправлять изменения на backend;
- умеет прикреплять файл/изображение к товару.

### Что делает раздел `Остатки`

- ищет товар по `SKU`;
- позволяет выбрать товар из списка;
- принимает фактическое количество;
- сравнивает ожидаемое и фактическое значение;
- сохраняет результат проверки в общее состояние приложения.

### Что делает раздел `Отчеты`

- формирует краткую статистику по инвентаризации;
- показывает количество:
  - проверенных позиций;
  - совпадений;
  - недостач;
  - избытков;
  - всех несоответствий;
- сохраняет отчет в текстовый файл.

## Ограничения и заметки

1. Основной рабочий клиент проекта сейчас — `WarehouseInventory.Desktop`.
2. `WarehouseInventory.Web` и `WarehouseInventory.Frontend` пока не подключены к backend полноценно.
3. Для web-проекта нужен SDK, поддерживающий `net10.0`.
4. В ряде файлов проекта встречаются проблемы с кодировкой русских строк; это влияет на читаемость исходников, но не меняет общую архитектуру.
5. Для корректной работы desktop-клиента backend должен быть запущен и доступен по ожидаемому адресу API.
